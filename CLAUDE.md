# MEE2024 — Modern Eddington Experiment

## Project Goal

Measure the gravitational deflection of starlight during solar eclipses, replicating and extending Eddington's 1919 experiment with modern equipment and analysis methods.

This is a fork of Andrew Smith's MEE2024 codebase, being developed in a new direction by Jesse Kinder.

## Two Tracks

See `plans.md` for full detail.

**Track 1 — Make It Work (Reproducible Research):** Community tool. TOML config files + CLI scripts + GUI "Load config" button. Validated against Bruns 2017 data. The Standard Pipeline (10 steps, with left/right calibration interpolation) is the target; all planned future MEE campaigns include eclipse-day calibration.

**Track 2 — Make It Right (Toolkit):** Personal research tool for Jesse. Modular, notebook-friendly, easy to extend. Begin only after Track 1 is working and validated.

## Branching Strategy

- `main` — Jesse's fork; Track 1 CLI work and Track 2 toolkit work
- `gui-improvements` — GUI-only changes for sharing with the MEE group repo; branch off `main` before any architectural changes

**Rule:** Non-GUI files (docs, configs, notes, tests, CLAUDE.md, skills) originate on `main` and are cherry-picked into `gui-improvements`. GUI-only commits (UI_handler.py, main.py GUI dispatch) stay on `gui-improvements` until that branch is ready to merge back. When `main` accumulates new non-GUI commits, cherry-pick them into `gui-improvements` to keep both branches current.

## Inter-Stage File Format

Each CLI script writes human-readable, descriptively-named output:

| Script | Output |
|--------|--------|
| `find-stars` | `{run}_centroids.csv` + `{run}_stack.fits` |
| `compute-distortion` | `{run}_distortion.toml` (coefficients + metadata) |
| `fit-data` | `{run}_deflection.csv` + diagnostic plots |

The existing ZIP-with-timestamp-hash format is legacy only.

## Project Skills

| Skill | Usage | Purpose |
|-------|-------|---------|
| `/explore-mee <module>` | e.g. `/explore-mee stacker_implementation` | Structured summary of a module: entry points, options keys, GUI coupling, porting notes |
| `/check-options <module>` | e.g. `/check-options distortion_fitter` | Extracts every `options` key used by a module and writes a skeleton TOML config to `examples/` |
| `/validate-pipeline [run]` | e.g. `/validate-pipeline bruns-2017` | Compares CLI output against golden reference output; checks star count, plate scale, pointing, roll, distortion coefficients, and final deflection |

## Validation (Golden Output)

Golden reference files live in `tests/golden/<run-name>/`. Each run directory must contain:
- Output files from the original GUI (centroid CSV, distortion coefficients, deflection results)
- `README.md` recording date, GUI version/git commit, config used, and input data path

To generate golden output: run the original GUI on a test dataset, copy outputs to `tests/golden/<run-name>/`, write the README.

CLI output to validate goes in `tests/output/<run-name>/`.

## Development Conventions

- New CLI scripts go in `scripts/`
- `configs/` — ready-to-run TOML configs for specific datasets (e.g. Station 1 sample data)
- `examples/` — skeleton TOML templates with all keys but no values (generated by `/check-options`)
- Config files use TOML format
- Use `tqdm` for progress bars; no `FreeSimpleGUI` imports in new code
- Output files use descriptive `{run}_*` names, not timestamp hashes
- Tests go in `tests/`; use real data from the test-data paths above (read-only)
